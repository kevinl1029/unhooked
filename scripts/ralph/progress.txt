## Codebase Patterns
- The spec file for instant conversations v2 is at `docs/specs/instant-conversations-spec.md` (was on feature/instant-conversations branch, checked out to ralph branch)
- Typecheck uses `npx vue-tsc --noEmit -p .nuxt/tsconfig.json` (requires .nuxt to be generated first via `nuxt prepare`)
- Existing codebase has pre-existing type errors (not related to this feature) — "typecheck passes" means no new errors introduced
- SQL migrations follow pattern in supabase/migrations/: use `ADD COLUMN IF NOT EXISTS`, always include COMMENTs, use `ON CONFLICT DO NOTHING` for idempotent inserts
- `#supabase/server` is a virtual module that cannot be resolved in the nuxt unit test environment — added resolve alias in `vitest.config.ts` pointing to `tests/__mocks__/supabase-server.ts` stub
- H3 server utilities (`defineEventHandler`, `getHeader`, `readBody`, `createError`) are globals in Nitro server files but NOT in the nuxt test environment — use direct logic extraction pattern for unit testing server handlers
- Unit testing server API handlers: extract the business logic into a helper function in the test file that accepts dependencies as parameters (consistent with `check-in-sender.test.ts` pattern)
- Unit testing server utility functions (no H3 globals): import directly and use `vi.mock()` + `vi.hoisted()` — no logic extraction needed (consistent with `precompute-opening.test.ts` pattern)
- Call order testing: push labels into a shared `callOrder: string[]` array from inside mock closures — all calls to `from()` share the same array since they close over it
- `vi.hoisted()` is required when referencing module-level variables inside `vi.mock()` factory functions
